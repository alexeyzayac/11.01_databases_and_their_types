### Задание 1. СУБД

### Кейс

Крупная строительная компания, которая также занимается проектированием и девелопментом, решила создать правильную архитектуру для работы с данными. Ниже представлены задачи, которые необходимо решить для каждой предметной области.

Какие типы СУБД, на ваш взгляд, лучше всего подойдут для решения этих задач и почему?

1.1. Бюджетирование проектов с дальнейшим формированием финансовых аналитических отчётов и прогнозирования рисков. СУБД должна гарантировать целостность и чёткую структуру данных.

1.1.* Хеширование стало занимать длительно время, какое API можно использовать для ускорения работы?

1.2. Под каждый девелоперский проект создаётся отдельный лендинг, и все данные по лидам стекаются в CRM к маркетологам и менеджерам по продажам. Какой тип СУБД лучше использовать для лендингов и для CRM? СУБД должны быть гибкими и быстрыми.

1.2.* Можно ли эту задачу закрыть одной СУБД? И если да, то какой именно СУБД и какой реализацией?

1.3. Отдел контроля качества решил создать базу по корпоративным нормам и правилам, обучающему материалу и так далее, сформированную согласно структуре компании. СУБД должна иметь простую и понятную структуру.

1.3.* Можно ли под эту задачу использовать уже существующую СУБД из задач выше и если да, то как лучше это реализовать?

1.4. Департамент логистики нуждается в решении задач по быстрому формированию маршрутов доставки материалов по объектам и распределению курьеров по маршрутам с доставкой документов. СУБД должна уметь быстро работать со связями.

1.4.* Можно ли к этой СУБД подключить отдел закупок или для них лучше сформировать свою СУБД в связке с СУБД логистов?

1.5.* Можно ли все перечисленные выше задачи решить, используя одну СУБД? Если да, то какую именно?

Приведите ответ в свободной форме.

#### Решение:

1.1. Реляционная СУБД подходит за счёт транзакционной целостности, формальной структуры данных и поддержки сложных запросов (например: PostgreSQL).

1.1.* Для ускорения хеширования можно использовать встроенные криптографические функции СУБД (например: pgcrypto в PostgreSQL). Если требуется ещё больше скорости, можно использовать специализированные API операционной системы (например: OpenSSL в Linux)

1.2. Для лендингов из-за гибкости и скорости хорошо подойдут документно-ориентированные нереляционная СУБД (например: MongoDB). CRM требует сложных отношений, транзакций и сложных запросов — там реляционная модель удобнее (например: PostgreSQL).

1.2.* Закрыть задачу одной СУБД можно с помощью PostgreSQL. Для лендингов: Использовать тип данных JSONB для хранения данных с форм. Это даёт гибкость и скорость, сравнимую с NoSQL, при этом все данные находятся в одной системе. Для CRM: Использовать классические реляционные таблицы для структурированных данных.

1.3. Структура компании иерархична, а знания связаны с определёнными узлами этой структуры. Графовая СУБД эффективно отображает иерархии и связи. (например: MongoDB). Также можно использовать уже выбранные реляционные СУБД (например: PostgreSQL) для создания простой, понятной структуры. 

1.3.* Лучше всего использовать PostgreSQL. Использовать тип данных JSONB для гибкого хранения метаданных и атрибутов материалов. Это будет проще для понимания и администрирования, чем внедрение отдельной графовой СУБД.

1.4. Это классическая задача для графовых СУБД. Они созданы для работы со сложными связями и алгоритмами обхода графов (например: ArangoDB)

### Задание 2. Транзакции

2.1. Пользователь пополняет баланс счёта телефона, распишите пошагово, какие действия должны произойти для того, чтобы транзакция завершилась успешно. Ориентируйтесь на шесть действий.

2.1.* Какие действия должны произойти, если пополнение счёта телефона происходило бы через автоплатёж?

Приведите ответ в свободной форме.

#### Решение:

2.1. Стандартное пополнение баланса (6 действий):

1. Инициация операции. Пользователь вводит номер телефона и сумму пополнения в интерфейсе приложения банка или платежного терминала.

2. Проверка и авторизация. Платежная система проверяет реквизиты, а также проводит авторизацию плательщика.

3. Списание средств. После успешной авторизации сумма пополнения списывается со счета банковской карты.

4. Передача данных оператору связи. Платежный шлюз формирует запрос и передает информацию об успешной оплате биллинговой системе оператора связи.

5. Зачисление средств на счет. Биллинг оператора связи зачисляет деньги на лицевой счет абонента.

6. Уведомление о результате. Система оператора связи отправляет SMS-уведомление пользователю об успешном пополнении баланса.

2.1.*  Пополнение через автоплатёж 

Разница только в 1 действие: 
1. Автоматическая инициация. В заранее установленную дату биллинговая система банка или оператора связи автоматически запускает процесс списания средств по предварительно сохраненному шаблону.

### Задание 3. SQL vs NoSQL

3.1. Напишите пять преимуществ SQL-систем по отношению к NoSQL.

3.1.* Какие, на ваш взгляд, преимущества у NewSQL систем перед SQL и NoSQL.

Приведите ответ в свободной форме.

#### Решение:

3.1. Преимущества SQL-систем по отношению к NoSQL:

1. ACID-транзакции (Полная гарантия целостности данных). Операция (например: пополнение баланса) выполняется либо полностью, либо не выполняется вовсе. В NoSQL часто используется eventual consistency (согласованность в конечном счёте), что может приводить к временным несоответствиям.

2. Мощный и универсальный SQL (Один стандартный язык для любых сложных запросов). Позволяет делать сложные выборки с соединениями, агрегацией и фильтрацией в одном запросе. В NoSQL языки запросов специфичны для каждой системы и часто менее выразительны.

3. Гибкость запросов (Можно задавать любые ad-hoc-вопросы к данным). Структура SQL позволяет получать ответы на вопросы, не заложенные при проектировании. NoSQL часто заточен под конкретные шаблоны доступа, и выход за их рамки ведёт к падению производительности.

4. Строгая схема данных (Данные всегда корректны и предсказуемы). Схема гарантирует, что данные соответствуют типам и ограничениям (например, email всегда в правильном формате). В NoSQL ответственность за целостность ложится на приложение, из-за этого данные могут стать неструктурированными и несогласованными.

5. Надёжность. SQL-системы имеют развитые инструменты для администрирования, резервного копирования, безопасности и огромные сообщества. В сравнении с ними экосистема многих NoSQL-баз всё ещё менее развита.
